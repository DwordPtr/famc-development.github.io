PHP Coding Standards.

Framework(s)

The use of frameworks on FAMC PHP projects should be evaluated on a case-by-case the accepted standard choices are Symfony.

Coding Style

FAMC PHP Coding Styles should strictly adhere to PHP-FIG (PHP Framework Interoperability Group) standards outlined in all accepted PSRs. The PSR-1 Basic Coding Standards and PSR-2 Coding Style Guide specifically address coding styles while PSR-0 address autoloading, namespacing and class naming (namespacing and class naming are likewise covered under PSR-1).

Classes

PHP classes must use the long "<?PHP" tag followed by a blank line. When present, namespace delcaration MUST follow the opening PHP tag and there MUST be one blank line after the namespace declaration. All use declarations MUST go after the namespace declaration and having one use keyword per declaration. There MUST be one blank line after the use block.
Classes should not have a closing "?>" tag.
Only one class is allowed per file.

Never use short tags: <? .. ?>. Short tags should be turned off in php.ini.

File Structure

The opening PHP ( <?PHP )tag should ALWAYS begin the file on line 1. Followed by a blank line a namespace declaration, if applicable, should always be on line 3 and be followed by a blank line. Any use statements required should then always begin on line 5 and be followed by a blank line. Class level PHPDoc should then begin on the next line and be followed immediately by the class declaration.

Class Names

Class names are to be declared in StudlyCaps (or ProudCamelCase). All classes written for PHP 5.3 and above should use namespaces. Classes written in 5.2 should follow pseudo-namespacing conventions outlined in PSR-1.

Class Members

ALL class members (properties and methods) should be first grouped by type putting constants first, followed by properties, then methods, then ordered by access modifier (public, protected then private) and last ordered by alphabetical order.

Constants

Constants MUST be declared in upper case with underscore separators.

Static variables

Static variables MUST be declared in upper case with underscore separators.

Properties

All properties should be declared as a descriptive variable name. Properties that are not constant nor static should be in camelCase.

Methods/Functions

Methods or Functions must be declared in camelCase.

Manipulation functions, functions which only modify the object and would normally be a void return should return the object to assist in method chaining -- unless the scope of the project prevents this.

Traits

Beginning with PHP 5.4 traits became an available language feature. Trait code should comply with coding standards classes except where noted below.

Trait Names

Trait names, just as classes, are to be declared in StudlyCaps ( or ProudCamelCase). All trait names should end with Trait for quick identification and distinction from other classes (e.g. WidgetTrait).

Control Structure

Spaces After Keywords

Control structure keywords (if, for, foreach, etc) should be followed by a space. This means a space should between the if and the opening parenthesis. This does NOT include method calls.

Example:
  • GOOD: if (true) { ... }
  • BAD: if(true) { ... }
  • GOOD: makeCoffee('sumatra');
  • BAD: makeCoffee ('sumatra');

Curly Braces

Curly braces should be on a new line for class and method declarations. Control stucture should be Egyptian Style (existing on the same line as the statement). Curly braces should always be included on if blocks regardless of line count - one line if statements should still include curly braces.

GOOD if (true) { // ... }

if (true) {
  // ...
  } else {
  // ...
  }

BAD if (true) // ...

if (true)
  // ...
  else
  // ...

Doc Blocks

Document everything. Follow proper PHPDoc standards (located: [http://www.phpdoc.org/]) PHPDoc blocks should exist just above the member or method they are describing. The Doc Block should include a brief but accurate description of the method, any arugments accepted, and if applicable what is returned.

Namespaces

All FAMC classes should fall under the top-level namespace of FAMC. The second level should be project name or codename. All levels of namespaces should be in StudlyCaps (or ProudCamelCase). In the interest maintaining PSR-0 and PSR-4 compliance packages should have a minimum of two levels of namespacing, a vendor level namespace (FAMC) and project (Blackbird), see example below. Maintaining separate namespaces allows for generic FAMC packages to be added via Composer and properly autoloaded.

\FAMC\Blackbird

Misc

Alignment

Pretty code is easy to read. Repetitive code spanning multiple lines should be aligned, this includes class property declarations, assignment opperations, or array member population. Property declarations should be aligned on name while assignment should align on equals and array member population should align on rockets.

Example:
// property declaration
  public      $name;
  public      $title;
  protected   $likesKoolAid;
  private     $koolAidFlavors;

  // Assignment
  $name         = "Miles Matheson";
  $title        = "Commander General of the Monroe Militia";
  $likesKoolAid = true;

  // Array member population
  $koolAidFlavors = array(
  'best'    => 'grape',
  'great'   => 'cherry',
  'just-ok' => 'fruit punch'
  );

Conditions

Yoda Conditions

When testing conditions always take care to not accidentally perform an assignment operation and Yoda conditions are a common techinique used to combat this -- these are not the PHP standard but are not discouraged. Yoda conditions are achieved by reversing your conditional statement e.g. if the condition is: 

"($y == '15')"

the Yoda condition would be written: 

"('15' == $y)"
Unnecessary Comparisons

When a function exists to handle a certain comparison, it should be used, as opposed to manually doing the comparison. An example would be comparing to null e.g. (is_null($z)) as opposed to ($z == null).

Magic Numbers

Do not use magic numbers. A magic number is a numeric constant embedded in code, without a constant definition. Any number except 0, 1 and 2 is considered magic.

if ($p->getX() < 10) /* BAD */

Use static final instead.

const WINDOW_XMAX = 10;
  // ...
  if ($p->getX() < self::WINDOW_XMAX) /* GOOD */

Even the most reasonable cosmic constant is going to change one day. You think there are 365 days per year? Your customers on Mars are going to be pretty unhappy about your silly prejudice. Make a constant

const DAYS_PER_YEAR = 365;

so that you can easily produce a Martian version without trying to find all the 365's, 364's, 366's, 367's, and so on, in your code.

(borrowed from Java Coding Standards)

Ternary Operator

Golden rule of Ternary Operator - "Use wisely; Use with caution"

A ternary operator should be use sparingly and only when truly needed. "Code like you cannot use it. If you find yourself complaining about not being able to use it then it is probably ok to use it there."

Example:

<?PHP

  namespace FAMC;

  use FAMC\Example\Lib\BaseClass;
  use FAMC\Example\Lib\LoanService;

  /**
  *  Franklin American Coder class
  *
  *  Used to write code. Some require coffee.
  *  Cupcake ipsum dolor. Sit amet cheesecake powder carrot cake marshmallow liquorice.
  *  Gingerbread gummi bears tart. Lemon drops topping macaroon caramels biscuit cupcake
  *  jelly beans icing pudding. Marzipan lollipop cheesecake brownie tart tiramisu lollipop.
  *
  *  Apple pie toffee chocolate caramels gummies cupcake pie. Wypas icing chocolate cake sugar
  *  plum cake chupa chups. Chocolate cake chocolate jelly oat cake cotton candy lemon drops
  *  dessert chocolate cake. Cookie wafer gingerbread dessert sweet roll marzipan cookie liquorice.
  *  Gummies gummi bears muffin donut tootsie roll gingerbread dessert powder brownie. Cotton candy
  *  powder cotton candy gingerbread topping. Wypas lollipop tiramisu sugar plum.
  *
  *  @package   FAMC\Example
  *  @author    Micah Breedlove <mbreedlove@franklinamerican.com>
  *  @copyright 2012 Franklin American Mortgage Company
  *
  */
  class Coder extends BaseClass
  {
  /**
  * NEEDS_SUGAR Coder Needs Sugar
  * @var string CONSTANT
  */
  const       NEEDS_SUGAR = true;

  /**
  * $coffee  Array used for coding
  *
  * @var    array
  * @access public
  */
  public      $coffee;

  /**
  * $project  Current working project
  *
  * @var string
  * @access public
  */
  public      $project;

  /**
  * $name  Current working project
  *
  * @var    string
  * @access protected
  */
  protected   $name;

  /**
  * $title   Job Title
  *
  * @var     string
  * @access  protected
  */
  protected   $title;

  /**
  * $likesCoffee If Coder likes Coffee
  *
  * @var     bool
  * @access  private
  */
  private     $likesCoffee;

  /**
  * Getter for Coffee it checks for the sugar need and adds
  * if necessary.
  *
  * @return array coffee array coder uses to write code
  */
  public function getCoffee()
  {

  if ($this->likesCoffee && self::NEEDS_SUGAR) {
  $this->addSugar();
  }

  return $this->coffee;
  }

  /**
  * Adds Sugar to Coffee
  *
  * @param int     $amount       Amount of sugar desired. Default is 1
  * @param string  $measurement  Measurement used to add sugar. Default is tspn
  *
  * @return \FAMC\Example\Coder
  */
  protected function addSugar($amount = 1, $measurement = 'tspn')
  {
  $this->coffee['sugar'] = array($amount, $measurement);

  return $this;
  }

  }
